# CLAUDE.md — Next.js 15 + SQLite SaaS Project

> Opinionated conventions for a production SaaS built with Next.js 15 App Router,
> SQLite (better-sqlite3 / Turso / libsql), and TypeScript.

## Stack

- **Framework:** Next.js 15, App Router (no Pages Router — remove if present)
- **Database:** SQLite via better-sqlite3 (local) or @libsql/client (Turso)
- **ORM:** Drizzle ORM (all DB access goes through Drizzle)
- **Auth:** NextAuth.js v5 / Auth.js
- **UI:** shadcn/ui + Tailwind CSS 4
- **Forms:** react-hook-form + zod
- **Package Manager:** pnpm (never npm/yarn)

## Project Structure

```
src/
├── app/              # Next.js App Router pages
│   ├── (auth)/       # Auth-related routes (login, register)
│   ├── (dashboard)/  # Authenticated routes
│   ├── api/          # Route handlers (no separate server dir)
│   └── layout.tsx    # Root layout
├── components/

│   └── features/     # Feature-specific components
├── db/
│   ├── schema/       # Drizzle schema files (one per domain)
│   ├── migrations/   # Auto-generated by drizzle-kit
│   └── index.ts      # DB connection + exports
├── lib/
│   ├── auth.ts       # Auth.js configuration
│   ├── utils.ts      # Shared utilities (cn(), formatDate(), etc.)
│   └── validations/  # Zod schemas (one file per domain)
├── hooks/            # React hooks (use-* prefixed)
├── actions/          # Server Actions (domain-grouped)
└── types/            # Shared TypeScript types
```

## Database & Migrations

- **NEVER** modify migration files directly. Migrations are auto-generated.
- Run `pnpm drizzle-kit push` for development, `pnpm drizzle-kit migrate` for production.
- Every schema file is one domain: `users.ts`, `projects.ts`, `subscriptions.ts`.
- All timestamps are `integer` (Unix ms) — don't mix with ISO strings.
- Use `text` for UUIDs: `id: text('id').primaryKey().$defaultFn(() => nanoid())`.

### Migration Rules
```sql
-- Anti-pattern (DON'T):
ALTER TABLE users ADD COLUMN stripe_customer_id TEXT;

-- Correct (DO):
-- 1. Edit src/db/schema/users.ts
-- 2. Run: pnpm drizzle-kit push
```

## SQL Conventions

- All queries go through Drizzle ORM — no raw SQL.
- Server Actions for writes; read queries in components.
- Use `cache()` (React.cache) for request-scoped data fetching.
- Always use prepared statements for dynamic values (Drizzle does this automatically).

```typescript
// ✅ Correct
const user = await db.select().from(users).where(eq(users.email, email));

// ❌ Wrong
await db.run(sql`SELECT * FROM users WHERE email = ${email}`);
```

## Component Patterns

- Server Components by default; Client Components only when needed (interactivity, hooks, browser APIs).
- Mark client boundary with `'use client'` — don't split unnecessarily.
- Props are typed with `interface`, never `type`.
- Use `cn()` from `@/lib/utils` for conditional classes, not template strings.

```typescript
// ✅ Correct
interface ButtonProps extends React.ButtonHTMLAttributes<HTMLButtonElement> {
  variant?: 'primary' | 'secondary' | 'ghost';
}
export function Button({ variant = 'primary', className, ...props }: ButtonProps) {
  return <button className={cn('px-4 py-2 rounded-md', variant === 'primary' && 'bg-blue-600', className)} {...props} />;
}
```

## Auth Conventions

- Session checked in middleware (`src/middleware.ts`) — redirect unauthenticated users.
- Current user data via a server-side helper: `getCurrentUser()` (cached per request).
- Protect API routes with `auth()` from Auth.js.
- Don't pass `session` as a prop — use `auth()` in layout or component directly.

## Error Handling

- Server Actions return `{ success: boolean; error?: string; data?: T }`.
- API routes return `NextResponse.json({ error: string }, { status: N })`.
- Use `error.tsx` files for route-level error boundaries, not try-catch everywhere.
- Log errors server-side with `console.error` (structured logging in production).

```typescript
// ✅ Server Action pattern
'use server';
export async function createProject(data: FormData) {
  try {
    const parsed = createProjectSchema.parse(Object.fromEntries(data));
    await db.insert(projects).values(parsed);
    revalidatePath('/dashboard');
    return { success: true };
  } catch (e) {
    if (e instanceof ZodError) return { success: false, error: e.errors[0].message };
    console.error('createProject failed:', e);
    return { success: false, error: 'Failed to create project' };
  }
}
```

## What We Don't Do (and Why)

1. **❌ No `any` types** — If you can't type it, you don't understand the data.
2. **❌ No barrel exports** (`index.ts` re-exports) — They break tree-shaking and cause circular deps.
3. **❌ No `useEffect` for data fetching** — Use Server Components, Server Actions, or React Query.
4. **❌ No inline styles** — Every visual decision goes through Tailwind classes or shadcn/ui theme tokens.
5. **❌ No database calls in components** — All DB access is in Server Actions or server-side helpers.
6. **❌ No Prisma** — We use Drizzle ORM. Prisma's SQLite support is second-class.
7. **❌ No `pages/` directory** — App Router only. If you find a Pages Router file, migrate it.
8. **❌ No environment variables in client code** — Use `NEXT_PUBLIC_*` prefix only for truly public values.

## Testing Philosophy

- Tests live next to the code they test: `lib/utils.test.ts`, `app/api/route.test.ts`.
- Server Actions are integration-tested with an in-memory SQLite database.
- Components are tested with Vitest + Testing Library (no Cypress for unit tests).
- E2E tests are a separate concern (Playwright or Cypress, project-level).
- Mock external APIs (Stripe, Resend, etc.) — don't call them in tests.
